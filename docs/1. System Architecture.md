# Low-Level System Architecture

This document provides a deep-dive into the technical internals, networking protocols, and execution model of the Lila Games Tic-Tac-Toe project.

---

## 1. Session Lifecycle & Authentication (HTTP/JWT)

The transition from a visitor to a connected player involves a shift from stateless HTTP to a stateful, authenticated session.

### A. The Authentication Handshake
When a user provides a `username` and `password`, the client initiates an **HTTP POST** request to `/v2/account/authenticate/email?create=true`. 
*   **Shadow Email Mapping:** The frontend silently maps `username` to `username@lila.games`.
*   **Server Processing:** Nakama's Go core receives the request, queries the PostgreSQL `users` table, and verifies the Bcrypt-hashed password.
*   **JWT Issuance:** Upon success, the server generates a **JSON Web Token (JWT)**. This token contains:
    *   `uid`: The permanent UUID of the user.
    *   `exp`: Expiration timestamp (default 2 hours).
    *   `username`: The player's display name.

### B. The WebSocket Upgrade (TCP/Stateful)
To enable real-time play, the client upgrades the connection:
1.  **Request:** Client sends an HTTP request with `Upgrade: websocket` and `Sec-WebSocket-Key`.
2.  **Authorization:** The JWT is passed as a bearer token in the headers or query string.
3.  **Session Assignment:** Nakama accepts the upgrade and assigns a **Session ID** (a transient UUID). This session is linked to the `user_id` but is unique to this specific TCP connection.

## 2. The Network Protocol (Protobuf & Envelopes)

Nakama does not send raw JSON over the wire. It uses **Google Protocol Buffers (Protobuf)** for efficiency, wrapped in a structured **Envelope**.

### A. The Envelope Structure
Every message (Match Data, Matchmaking requests, etc.) is wrapped in a Protobuf Envelope:
*   **`cid` (Correlation ID):** A string used to match requests with responses.
*   **Payload:** The actual message (e.g., `MatchDataSend`).

### B. Serialization Flow
When a player makes a move at index 4:
1.  **Serialization:** The React client converts `{index: 4}` into a **UTF-8 Byte Array**.
2.  **Framing:** This array is placed inside a `MatchDataSend` Protobuf object.
3.  **Transmission:** The binary frame is sent over the WebSocket (OpCode `0x2` for Binary).

## 3. Matchmaking Pool & Spawning (In-Memory)

The Matchmaking system is an in-memory "Waiting Room" managed by Nakama's Go core.

1.  **Pool Entry:** `socket.addMatchmaker` sends a `MatchmakerAdd` request. Nakama adds the player's `session_id` to a specialized memory grid.
2.  **Matching Logic:** The engine periodically scans the grid for players with matching criteria (e.g., 2 players, wildcard query `*`).
3.  **The Runtime Hook:** When Alice and Bob are matched, the Go core "invokes" the JavaScript Virtual Machine (QuickJS).
4.  **Match Spawning:** The server runs the `matchmakerMatched` function in `hooks.ts`:
    *   It calls `nk.matchCreate('tictactoe')`.
    *   This allocates a **Match Handler State** in the server's RAM.
    *   The `match_id` is generated and returned to the paired players.

---

## 4. The Authoritative Execution Model (QuickJS & Ticks)

Your server logic (`match.ts`) runs inside a **QuickJS Virtual Machine** embedded in the Nakama binary.

### A. The Tick Rate
The match is configured with `tickRate: 1`. This means the server "wakes up" and runs the `matchLoop` exactly **once per second**.

### B. Input Buffering
If Player A sends a move at 0.1s and Player B sends a move at 0.5s, the server does **not** process them immediately. It buffers these messages in a queue until the next tick (1.0s).

### C. The Match Loop Execution
1.  **Injection:** At the start of the tick, the Go engine "injects" the current `state` object and the buffered `messages` into the JS environment.
2.  **Logic Phase:** Your `matchLoop` iterates through the messages. It performs **Validation** (Is it the player's turn?) and **Mutation** (Update the board array).
3.  **Broadcast Phase:** Any messages queued via `dispatcher.broadcastMessage` are collected.
4.  **Serialization:** The modified `state` is snapshotted, and the broadcast messages are serialized into binary Protobuf envelopes and sent to the clients.

---

## 5. State Synchronization & Draw Detection

The server enforces the "Source of Truth" through the `UPDATE` OpCode.

*   **Diffing:** The client does not calculate the game state; it only renders what the server sends.
*   **Win/Draw Logic:** The server calculates wins by checking the 8 possible winning combinations (3 rows, 3 cols, 2 diagonals) against the `board` array.
*   **Finality:** When a win or draw is detected, the server sends the `GAME_OVER` OpCode. This message includes the **final board state** to ensure the client's last frame is accurate before the match handler is destroyed.

---

## 6. Cleanup & Resource Management

*   **Graceful Exit:** When a player leaves, the `matchLeave` function is called, allowing the server to decide if the game should end immediately or wait for a reconnect.
*   **Deallocation:** Once all players leave or the game ends, Nakama destroys the Match Handler's memory context and stops the tick timer, ensuring zero resource leakage.
